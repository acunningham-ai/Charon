Skills & Hooks

Behaviour you can read. Guardrails you can't talk past.

Charon is built from three kinds of parts — rules that shape how the model behaves, skills it can run on request, and hooks: deterministic code the harness runs on its own, outside the model's discretion.

The three layers

Doctrine, capability, and enforcement — kept separate.

Each layer does one job. Rules teach the model how to work; skills give it things to do; hooks enforce the lines that shouldn't depend on the model choosing well.

Layer 01 · doctrine

Rules

Markdown that shapes behaviour

Four always-fire rules run every turn — ask when unsure, tag confidence, save facts on mention, load memory at session start. Seven more inject the right doctrine only when the path or keyword fires, so guidance arrives in context, not as noise.

Skills

Things the model can run

Thirty-nine slash commands — reporting and governance prep, an independent security-review suite, the research-to-compose pipeline, knowledge-graph queries, and hygiene checks. Each declares a minimal tool set, so a skill only holds the permissions it actually needs.

Layer 03 · enforcement

Hooks

Code the harness runs itself

Ten Python hooks fire on real events — a prompt arriving, a tool about to run, a scheduled job failing. They don't ask the model's permission. A write outside the allowlist is denied whether or not the model meant well.

validate-write-pathdeny-destructiveload-rules

Why hooks matter

The guardrail runs even when the model is wrong.

A rule is guidance — powerful, but the model still chooses whether to follow it. A hook is deterministic code wired to a tool event. It runs before the action lands, returns a verdict, and can't be reasoned around. That's the difference between "we told it not to" and "it couldn't."

Every hook is Python you can open in scripts/hooks/. No black box — the enforcement is as auditable as the behaviour.

Named agents

A small crew of standing roles — each with one job, and its own memory.

Some agents are named functional seats, not one-off commands. They carry work across sessions, keep their own working notes, and hand off to one another. They're roles — a research analyst, a writer — never a roleplay of you.

Prometheus

/prometheus

The research seat · the Titan of forethought

A standing analyst with a persistent ledger of your research beats. Each run it researches the top threads and writes a prioritised digest with framed angles — so scattered reading becomes a daily "so what." It marks which findings are worth writing about and frames the use case, ready to hand to Calliope. An optional beat pulls the CISA known-exploited-vulnerabilities catalogue, ranked for triage.

Surfaces and steers — never acts on its own

Calliope

/calliope

The writing seat · the Muse of eloquence

Composes in your voice across modes — post, stakeholder bulletin, tweet, email — taking a Prometheus angle or a raw topic to a finished draft. It loads your voice profile and anchors first, so the draft sounds like you, not a generic model. Drafting is the bottleneck; an AI that can send is a liability. So Calliope drafts only.

Drafts only — never sends, posts, or emails

Named seats are functional roles, not personas — a documented anti-pattern is having an agent roleplay your identity. A delivery seat (Hermes) is designed but not yet shipped.

The automation between them

Research → compose → deliver. Hand off, don't hand over.

The seats aren't isolated. They form a pipeline: one frames the work, the next drafts it, and — by design — a human stays on the consequential edge. Nothing goes out without you.

Prometheus keeps a cross-session ledger you steer; it frames a content-worthy angle and hands it to Calliope, who drafts it in your voice. You approve. A delivery seat would then send — but that seat is deliberately unbuilt until the guardrails around an agent that can send are ready.

Cross-session handoff

The seats don't lose the thread between conversations — Prometheus's ledger and digest persist, so today's framed angle is still there when you sit down to write.

Human on the edge

The pipeline automates the work, not the decision. Research and drafting run freely; the moment something would leave your control, it stops for you.

Security review, built in

A security reviewer that never clocks off.

The secure-code rule fires the moment you touch code, and three OWASP-grounded lenses can review it in parallel — then /fp-check throws out the findings it can't reproduce. There's also a capability that audits the AI installation itself: its config, plugins, and the dependencies you pull.

/secure-code-review

Baseline secure-coding lens plus C-1…C-8 coverage.

/owasp-llm-review

OWASP Top 10 for LLM Applications (2025), LLM01–LLM10.

/owasp-agentic-review

OWASP Agentic AI Security (2026), ASI01–ASI10.

/fp-check

Re-reads each finding at file:line — withdraws what it can't reproduce.

Cerberus suite

Read-only audit of the AI installation: config, plugins, MCP servers, dependencies. Opt-in, never silently enabled.
