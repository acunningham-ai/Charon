Skills & Hooks

Behaviour you can read. Guardrails you can't talk past.

Charon is built from three kinds of parts — rules that shape how the model behaves, skills it can run on request, and hooks: deterministic code the harness runs on its own, outside the model's discretion.

The three layers

Doctrine, capability, and enforcement — kept separate.

Each layer does one job. Rules teach the model how to work; skills give it things to do; hooks enforce the lines that shouldn't depend on the model choosing well.

Layer 01 · doctrine

Rules

Markdown that shapes behaviour

Four always-fire rules run every turn — ask when unsure, tag confidence, save facts on mention, load memory at session start. Ten more inject the right doctrine only when the path or keyword fires, so guidance arrives in context, not as noise.

Layer 02 · capability

Skills

Things the model can run

Thirty-nine slash commands — reporting and governance prep, an independent security-review suite, knowledge-graph queries, day-and-week runners, and hygiene checks. Each declares a minimal tool set, so a skill only holds the permissions it actually needs. (The named agents below are invoked as commands too — but they're standing roles, not one-off skills.)

/quarterly-report-prep/triage-inbox/secure-code-review/vault-query
Layer 03 · enforcement

Hooks

Code the harness runs itself

Ten Python hooks fire on real events — a prompt arriving, a tool about to run, a scheduled job failing. They don't ask the model's permission. A write outside the allowlist is denied whether or not the model meant well.

validate-write-pathdeny-destructiveload-rules

Why hooks matter

The guardrail runs even when the model is wrong.

A rule is guidance — powerful, but the model still chooses whether to follow it. A hook is deterministic code wired to a tool event. It runs before the action lands, returns a verdict, and can't be reasoned around. That's the difference between "we told it not to" and "it couldn't."

Every hook is Python you can open in scripts/hooks/. No black box — the enforcement is as auditable as the behaviour.

Security review, built in

A security reviewer that never clocks off.

The secure-code rule fires the moment you touch code, and three OWASP-grounded lenses can review it in parallel — then /fp-check — a false-positive check — throws out the findings it can't reproduce. There's also a capability that audits the AI installation itself: its config, plugins, and the dependencies you pull.

/secure-code-review

Baseline secure-coding lens plus C-1…C-8 coverage.

/owasp-llm-review

OWASP Top 10 for LLM Applications (2025), LLM01–LLM10.

/owasp-agentic-review

OWASP Agentic AI Security (2026), ASI01–ASI10.

/fp-check

False-positive gate — re-reads each finding at file:line and withdraws what it can't reproduce.

Cerberus suite

Read-only audit of the AI installation: config, plugins, MCP servers, dependencies. Opt-in, never silently enabled.
